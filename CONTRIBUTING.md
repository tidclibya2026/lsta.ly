# دليل المساهمة في LSTA

هذا المستودع مشروع حكومي منضبط. لا تُقبل التعديلات المباشرة على `main`.

## سير العمل

1. حدّث نسختك من `main`.
2. أنشئ فرعًا باسم واضح:
   - `feature/...`
   - `fix/...`
   - `docs/...`
   - `release/...`
3. نفّذ التعديل ضمن نطاق واحد قابل للمراجعة.
4. شغّل الاختبارات والفحوص المطلوبة.
5. افتح Pull Request يوضح الغرض، الأثر، الاختبارات، ومخاطر البيانات.
6. لا يتم الدمج قبل نجاح CI والمراجعة.

## قواعد البيانات والملفات

- لا ترفع ملفات `.env` أو كلمات المرور أو مفاتيح API.
- لا ترفع ملفات KML/KMZ أو الصور أو البيانات الخام إلا بتفويض مكتوب.
- لا تعدّل ملفات المصدر الأصلية داخل مسار الاستيراد.
- استخدم Alembic لأي تغيير في قاعدة البيانات.
- لا تستخدم `create_all` بدل migrations.
- لا تنقل بيانات من Staging إلى السجل الوطني دون Workflow مراجعة موثق.

## فحوص Frontend

```powershell
cd apps\atlas-government
npm test
npm run typecheck
npm run build
```

## فحوص Backend

```powershell
cd services\atlas-api
pytest
ruff check .
python -m compileall app
```

## رسائل Commit

استخدم نمطًا واضحًا:

```text
feat: إضافة محرك البحث الوطني
fix: منع تكرار الترقية
refactor: فصل خدمة العلاقات المكانية
test: إضافة اختبار صلاحيات المراجع
docs: تحديث دليل التشغيل
```

## متطلبات Pull Request

يجب توضيح:

- المشكلة أو القدرة التشغيلية المطلوبة.
- الملفات والوحدات المتأثرة.
- أثر التغيير على Staging والسجل الوطني.
- نتيجة الاختبارات والبناء.
- أي migration جديدة.
- ما إذا كان التغيير يؤثر على النشر العام أو Visit Libya.
- لقطات شاشة عند تعديل الواجهة.

## مبدأ الفصل بين المهام

لا يجوز للمحرر اعتماد تعديله بنفسه. يجب الحفاظ على فصل الإدخال والمراجعة والاعتماد والنشر.
